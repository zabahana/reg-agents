## Appendix A — Validation Agent findings

### Findings for Model FRAUD-XGB-GNN-001

#### 1. Conceptual Soundness
**GAP**: The model documentation lacks a comprehensive explanation of the GNN-enhanced XGBoost methodology, particularly how the graph embeddings are generated and integrated with tabular features. This could lead to misunderstandings regarding the model's decision-making process.

**RECOMMENDATION**: Enhance the methodology section to include detailed descriptions of the GraphSAGE algorithm, how embeddings are created, and the rationale for their inclusion in the model. This aligns with the need for documented rules and thresholds as per [ffiec_bsa_aml.md].

#### 2. Data Quality
**GAP**: While the model uses a substantial dataset, there is no mention of data quality checks or validation processes for the training data. Given the class imbalance (0.3% positive), it is crucial to ensure that the data is clean and representative.

**RECOMMENDATION**: Implement a robust data quality validation process that includes checks for missing values, outliers, and data consistency. Document these processes in the model card to comply with the expectations outlined in [ffiec_bsa_aml.md].

#### 3. Testing/Outcomes Analysis
**GAP**: The outcomes analysis is limited to a 6-month period, which may not capture long-term performance trends. Additionally, there is no mention of disparate-impact testing across customer segments, which is critical for fair lending compliance.

**RECOMMENDATION**: Extend the outcomes analysis to cover a longer time frame and include disparate-impact testing to ensure compliance with fair lending regulations [ecoa_reg_b.md]. Document the results and any corrective actions taken.

#### 4. Ongoing Monitoring
**GAP**: The model currently lacks automated drift-triggered retraining mechanisms. While PSI and AUC are monitored, the absence of a systematic approach to retraining could lead to performance degradation over time.

**RECOMMENDATION**: Develop and implement an automated monitoring system that triggers retraining based on significant feature drift or performance drop. This aligns with the need for ongoing performance monitoring as stated in [ffiec_bsa_aml.md].

#### 5. Fair-Lending/ECOA Exposure
**GAP**: Although age and ZIP code features are excluded to mitigate fair-lending proxy risks, there is no comprehensive fair-lending review documented. The potential for other features to act as proxies for protected classes has not been assessed.

**RECOMMENDATION**: Conduct a thorough fair-lending review of all model features to ensure that none act as proxies for protected classes. Document the findings and any adjustments made to the model features to comply with [ecoa_reg_b.md].

### Summary of Recommendations
1. **Enhance Methodology Documentation**: Provide detailed descriptions of the GNN-enhanced XGBoost methodology.
2. **Implement Data Quality Checks**: Establish and document data quality validation processes.
3. **Extend Outcomes Analysis**: Broaden the outcomes analysis period and include disparate-impact testing.
4. **Automate Ongoing Monitoring**: Create an automated system for drift detection and retraining.
5. **Conduct Fair-Lending Review**: Assess all features for potential proxy risks and document findings.

By addressing these gaps and implementing the recommendations, the model can better align with regulatory expectations and improve its robustness and fairness in fraud detection.

## Appendix C — Retriever Agent regulatory context

The regulatory requirements for validating and monitoring a model used for 'Real-time card transaction fraud scoring' include the following points:

1. **Model Risk Management**: The model used for fraud scoring should adhere to model risk management principles as outlined in OCC Bulletin 2011-12 and Federal Reserve SR 11-7. This guidance is principles-based and should scale with the bank's size, complexity, and model use [occ_2011-12_model_risk.md, Relationship to SR 11-7].

2. **Documented Rules and Thresholds**: The automated transaction-monitoring and fraud systems must have documented rules and thresholds. This includes tuning and testing (above-the-line/below-the-line testing), data-quality validation, and ongoing performance monitoring [ffiec_bsa_aml.md, Transaction Monitoring Systems].

3. **Governance Approval for Threshold Changes**: Any changes to the thresholds used in the model require governance approval, ensuring that there is oversight and accountability in the model's operation [ffiec_bsa_aml.md, Transaction Monitoring Systems].

4. **Fair-Lending Review**: Fair-lending risk should be integrated into the model risk management process. This includes conducting fair-lending reviews at the development stage, during validation, and in ongoing monitoring, with results reported to governance committees [fair_lending_exam.md, Governance Expectations].

5. **Feature Selection and Monitoring**: The selection of model features should be scrutinized to ensure they do not act as proxies for protected classes. This includes conducting fair-lending reviews as part of the feature selection and monitoring process [ecoa_reg_b.md, Model Feature Governance].

6. **Validation and Explainability**: The models used for fraud detection must be validated, explainable, and tuned to balance detection rates against false positives, ensuring that the models are effective and compliant with regulatory expectations [udaap_bsa_aml.md, BSA/AML — Bank Secrecy Act / Anti-Money Laundering].

These points collectively outline the necessary regulatory framework for validating and monitoring a model used for real-time card transaction fraud scoring.
