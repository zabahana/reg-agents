### Findings for Model FRAUD-XGB-GNN-001

#### 1. Conceptual Soundness
**GAP:**
- The model's methodology is innovative but lacks comprehensive documentation on the theoretical underpinnings of the GNN-enhanced XGBoost approach. There is limited explanation of how the graph embeddings specifically contribute to fraud detection effectiveness.

**RECOMMENDATION:**
- Enhance the documentation to include a detailed explanation of the GNN methodology, including how it improves upon traditional XGBoost approaches. Reference relevant literature to support the choice of model architecture and its expected benefits in fraud detection.

#### 2. Data Quality
**GAP:**
- While the model uses a substantial dataset, there is no mention of data quality checks or validation processes for the training data. The model documentation does not specify how data anomalies or inaccuracies are handled.

**RECOMMENDATION:**
- Implement and document a robust data quality validation process that includes checks for missing values, outliers, and data consistency. Ensure that this process is aligned with the requirements outlined in [ffiec_bsa_aml.md] regarding data-quality validation.

#### 3. Testing/Outcomes Analysis
**GAP:**
- The outcomes analysis is limited to a 6-month period, which may not capture long-term performance trends. Additionally, disparate-impact testing across customer segments has not been documented.

**RECOMMENDATION:**
- Extend the outcomes analysis to cover a longer time frame to assess model stability and performance over time. Conduct and document disparate-impact testing to ensure compliance with fair-lending regulations, as highlighted in [ecoa_reg_b.md].

#### 4. Ongoing Monitoring
**GAP:**
- The current monitoring approach lacks automated drift detection and retraining mechanisms. The manual review process may lead to delays in addressing model performance issues.

**RECOMMENDATION:**
- Develop and implement automated monitoring tools that trigger alerts for significant model drift or performance degradation. Establish a governance framework for timely retraining based on these alerts, in line with the expectations set forth in [ffiec_bsa_aml.md].

#### 5. Fair-Lending/ECOA Exposure
**GAP:**
- Although age and ZIP code features are excluded to mitigate fair-lending risks, there is no comprehensive fair-lending review of the remaining features to ensure they do not inadvertently act as proxies for protected classes.

**RECOMMENDATION:**
- Conduct a thorough fair-lending review of all model features to ensure compliance with [ecoa_reg_b.md]. This should include an analysis of potential proxies and an assessment of how feature selection may impact different demographic groups.

### Summary of Recommendations
1. **Enhance Model Documentation**: Provide detailed theoretical explanations of the GNN-enhanced XGBoost methodology.
2. **Implement Data Quality Checks**: Establish a rigorous data validation process to ensure the integrity of training data.

## Appendix C — Retriever Agent regulatory context

The regulatory requirements for validating and monitoring a model used for 'Real-time card transaction fraud scoring' include the following points:

1. **Model Risk Management**: The model should be subject to model risk management principles, which are outlined in OCC Bulletin 2011-12 and Federal Reserve SR 11-7. This includes ensuring that the management practices scale with the bank's size, complexity, and model use [occ_2011-12_model_risk.md, Relationship to SR 11-7].

2. **Documentation and Testing**: The automated transaction-monitoring and fraud systems must have documented rules and thresholds, undergo tuning, and include both above-the-line and below-the-line testing. Additionally, data-quality validation and ongoing performance monitoring are essential [ffiec_bsa_aml.md, Transaction Monitoring Systems].

3. **Governance Approval for Threshold Changes**: Any changes to thresholds in the model require governance approval, ensuring that there is oversight and accountability in the model's operation [ffiec_bsa_aml.md, Transaction Monitoring Systems].

4. **Fair-Lending Review**: Fair-lending risk should be integrated into the model risk management process, which includes conducting fair-lending reviews at the development stage, during validation, and in ongoing monitoring. The results of these reviews should be reported to governance committees [fair_lending_exam.md, Governance Expectations].

5. **Feature Governance**: It is important to ensure that model features do not act as proxies for protected classes. This requires careful feature selection and monitoring, which should also include a fair-lending review [ecoa_reg_b.md, Model Feature Governance].

6. **Explainability and Tuning**: The models used for fraud detection must be validated, explainable, and tuned to effectively balance detection rates against false positives [udaap_bsa_aml.md, BSA/AML — Bank Secrecy Act / Anti-Money Laundering].

These points collectively outline the necessary regulatory framework for validating and monitoring a model used for real-time card transaction fraud scoring.
