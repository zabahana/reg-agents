## Appendix A — Validation Agent findings

### Findings for Model Validation of Card Transaction Fraud Detector (FRAUD-XGB-GNN-001)

#### 1. Conceptual Soundness
**GAP:** The model documentation lacks a comprehensive explanation of the GNN-enhanced XGBoost methodology, particularly how the GraphSAGE embeddings are integrated and their impact on model performance. This is crucial for understanding the model's assumptions and limitations.

**RECOMMENDATION:** Enhance the documentation to include detailed descriptions of the GNN architecture, the rationale for its use, and how it interacts with the XGBoost classifier. This aligns with the need for documented rules and thresholds as per [ffiec_bsa_aml.md].

#### 2. Data Quality
**GAP:** While the model uses a substantial dataset (240M rows), the handling of class imbalance is only partially addressed. The documentation does not specify how the model performs with respect to different segments of the data, particularly in terms of data quality and representativeness.

**RECOMMENDATION:** Implement a more robust data quality validation process that includes segment-level analysis to ensure that the model performs adequately across various customer demographics. This is in line with the expectations for data-quality validation in [ffiec_bsa_aml.md].

#### 3. Testing/Outcomes Analysis
**GAP:** The outcomes analysis is limited to a 6-month period, which may not capture long-term performance trends. Additionally, there is no mention of backtesting against historical data beyond this period.

**RECOMMENDATION:** Extend the outcomes analysis to cover a longer time frame and conduct backtesting to validate model performance over time. This is essential for high-risk models as stated in [model_validation_standards.md].

#### 4. Ongoing Monitoring
**GAP:** The current monitoring approach relies on manual reviews and does not include automated drift detection or retraining mechanisms. This could lead to delays in addressing model performance degradation.

**RECOMMENDATION:** Develop an automated monitoring system that triggers retraining based on performance drift. This should include regular updates to the model based on real-time data feedback, as recommended in [ffiec_bsa_aml.md].

#### 5. Fair-Lending/ECOA Exposure
**GAP:** Although age and ZIP code features are excluded to mitigate fair-lending proxy risk, there is no evidence of disparate-impact testing across customer segments. This raises concerns about potential biases that could affect fair lending compliance.

**RECOMMENDATION:** Conduct a thorough fair-lending review during model development and validation, including disparate-impact testing across various customer segments. Results should be reported to governance committees as per [fair_lending_exam.md].

### Summary of Recommendations
1. **Enhance Methodology Documentation:** Provide detailed descriptions of the GNN-enhanced XGBoost methodology.
2. **Robust Data Quality Validation:** Implement segment-level analysis to ensure model performance across demographics.
3. **Extend Outcomes Analysis:** Conduct longer-term outcomes analysis and backtesting.
4. **Automated Monitoring System:** Develop automated drift detection and retraining mechanisms.
5. **Conduct Disparate-Impact Testing:** Integrate fair-lending reviews and testing into the model validation process.

By addressing these gaps and implementing the recommendations, the model can better align with regulatory expectations and improve its robustness and fairness in fraud detection.

## Appendix C — Retriever Agent regulatory context

The regulatory requirements for validating and monitoring an AI-based fraud model include the following points:

1. **Validation Responsibility**: Banks are responsible for validating vendor or third-party AI models. They must obtain sufficient information to evaluate the model's conceptual soundness, monitor its performance, and understand its limitations, even if the model is proprietary. [sr11-7_model_risk.md, Vendor and AI/ML Models]

2. **Model Risk Management**: Automated transaction-monitoring and fraud systems must adhere to model risk management practices. This includes having documented rules and thresholds, conducting tuning and testing (both above-the-line and below-the-line), validating data quality, and ongoing performance monitoring. Any changes to thresholds require governance approval. [ffiec_bsa_aml.md, Transaction Monitoring Systems]

3. **Fair-Lending Review**: It is essential to integrate fair-lending risk into model risk management. This involves conducting fair-lending reviews during the development, validation, and ongoing monitoring phases of the model, with results reported to governance committees. [fair_lending_exam.md, Governance Expectations]

4. **Bias Monitoring**: Institutions must document the drivers of decisions made by automated models that affect transactions or accounts. They should also support human review of high-impact actions and monitor for bias across different customer segments. [udaap_bsa_aml.md, Model Explainability for Consumer-Facing Decisions]

These points highlight the importance of thorough validation, performance monitoring, fair-lending considerations, and bias monitoring in the context of AI-based fraud models.
